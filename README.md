# MacCare — macOS Agentic Health Assistant

> An agentic AI terminal assistant that monitors macOS system health using a **Supervisor-Worker** multi-agent architecture, powered by **Llama 3.2 via Ollama**.

---

## Features

| Category | What it checks |
|---|---|
| 💾 Storage | Disk usage, largest folders, big files |
| 🗑️ Cache | All cache directories with sizes |
| 🧹 Cleanup | Safe-to-delete files with exact commands |
| 🧠 Memory | RAM usage, swap, memory pressure |
| ⚡ CPU | Overall & per-core usage, top processes |
| 🔋 Battery | Charge %, cycles, capacity, draining apps |
| 🖥️ System | macOS version, chip, RAM, uptime |
| ⚡ Watchdog | Background battery & CPU spike monitor |

## Quick Start

### 1. Install dependencies

```bash
cd /Users/bercaakbayir/Desktop/MacCare
pip install -r requirements.txt
```

### 2. Install & start Ollama (for LLM mode)

```bash
# Install Ollama
brew install ollama

# Pull Llama 3.2 model
ollama pull llama3.2

# Start Ollama server (in a separate terminal)
ollama serve
```

### 3. Run MacCare

```bash
# LLM mode (requires Ollama running)
python app.py

# Offline mode (no LLM — rule-based, always works)
python app.py --offline

# Custom model
python app.py --model llama3.2:1b
```

```
        ├── MemoryAgent  → tools/memory.py

        └── EnergyWatchdog (Background Thread)
```

### Multi-Agent ReAct Flow

```
User Query
    │
    ▼
Supervisor (Llama 3.2)
    │  Thought: What tools do I need?
    │
    ├─→ StorageAgent  (df, du, find)
    ├─→ MemoryAgent   (psutil, vm_stat)
    ├─→ SystemAgent   (pmset, system_profiler)
    │
    │  Observation: [tool results JSON]
    │
    ▼
Supervisor synthesises → Rich-formatted answer
```

### Agent Orchestration & Agentic Engine

MacCare utilizes a robust **ReAct (Reason → Act → Observe → Reflect)** engine structured via a precise Supervisor-Worker pattern:

1. **Supervisor Agent (`MacCareOrchestrator`)**: The core "brain" of the application. It receives user inputs, prepends system prompts and strict policies from the `SafetyAuditor`, and communicates with Llama 3.2 via Ollama. It utilizes native LLM **Tool Calling** (JSON structuring).
2. **Reasoning & Tool Dispatch**: The Supervisor passes the `TOOL_REGISTRY` mapping to the LLM. If the user asks a question requiring systemic data (e.g., "what files can I delete?"), the LLM generates tool execution intents. The orchestrator intercepts these, dynamically filters arguments against known schema structures (shutting down LLM parameter hallucinations), and routes them to specialized Sub-Agents.
3. **Sub-Agents (Workers)**:
    - **`StorageAgent`**: Executes disk usage checks and folder mapping tasks natively.
    - **`MemoryAgent`**: Interfaces with RAM and CPU statistics via OS system layers.
    - **`SystemAgent`**: Reports back on battery cycles and hardware runtime status.
4. **Observe & Reflect (The Loop)**: The results generated by Sub-Agents are audited and appended with semantic "Health Evaluations" by the `SafetyAuditor` (to give statistical markers meaning, like '95% RAM = Critical'). This data is fed *back* to the LLM. The LLM evaluates if its data collection is complete or if further tools should be triggered.
5. **Final Synthesis**: Once the LLM determines it has sufficient data, it closes out the ReAct loop and streams all observational data into a polished, conversational response to the user.
6. **Graceful Fallback**: If Ollama goes offline, the Supervisor instantly downgrades to a zero-downtime, hardcoded rule-based router that executes matching tools instantly.
